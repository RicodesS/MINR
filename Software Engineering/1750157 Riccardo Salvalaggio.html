REST Web service is an architectural-style for developing web services that emphasizes WWW as an open and standard application.
Since Web became more and more famous and used, Firewalls have started to let port 80 always open to permit a secure and easy communication and exchanging of messages between Clients and Web.


A famous metaphor is the one that defines:
URI ~ : nouns
GET/POST etc. : verbs, so action to perform. 

REST is the perfect example of use of API (interfaces provided by servers and discoverable by requestors).
REST uses HTTP (a standard communication protocol for communication over the web) as standard for communications and uses also its error codes, such as 404 for page not found.
The use that a REST Web service does of HTTP signs its "Maturity", in fact there are 4 level of maturity of  REST:
0 - HTTP as a tunnel
1 - Resources
2 - self-describing messages
3 - Hypermedia

So a Web service is called "RESTful" only if it has a maturity level of 3 i.e. uses hypermedia embedded documents in exchanging messages.

REST has also some design constraints e.g. uniform interface or stateleness interaction, that is also a disadvantage of REST togheter with the too much dependence with HTTP.
The advantages are: World-wide popularity, firewall friendleness and simplicity.

SCRUM is an agile processes used in IT & not-IT companies, its principal objective is to produce the most amount of business possible, reducing the time of release
It interfaces with the relay race: now we concentrate most on how to release sw in less time, instead of concentrate on the reliability and flexibility of the software.
The SCRUM process wants some steps:
Product backlog - Sprint backlog - Sprint - Daily scrum - End (Runnable SW)
Every of these steps are described in the SCRUM Framwork divided in: Roles, activities and artifacts.

SCRUM Roles:
Product Owner: "The person that invests in the project", he has to decide what will be the input and output of the project, he set the requirements and the goals, and also the release date and if to accept or reject the work.
ScrumMaster: Is the guy "expert" of SCRUM , he has to follow the sw processes to ensure that the team is following the SCRUM principles; he has also to shield the team from the external attacks.
Team: Is composed by 5-7 people, the components have to be self-managed and cross-functional. The composition of the team cannot change during a sprint.

SCRUM Activities:
Sprint planning: Now we have to decide which goal we want to implement during the sprint, re-prioritising the ones not completed, and decide how to continue.
Sprint: is a period of 2-4 weeks in which the team develop the stories decided
Daily Scrum: is an activity that the team has to do every day to discuss about the project.

SCRUM Artefacts:
Product backlog: In this phase requirements of the project have to be decided and prioritised.
Sprint backlog: We have to decide how to continue, look at the stories not finished and re-prioritise them.
Sprint review: Finished the sprint, we have to review the sprint, so talk with the all team and managers about the sprint.
Sprint retrospective: Before starting a new sprint we have to look at what we have done until now and decide if continue or stop.

To evaluate an estimation in terms of time and effort in a software development process we use Function Point.
At the base of this method there is a dimension metric: LOC or SLOC (Standard LOC), that indicates the number of lines of code estimated for a project, it can be seem an easy way of evaluaton, but it involve so many parameters, such as the programming language, the technologies, paradigms etc.

Funciton points uses 5 types of functionalities to separate the requirements:
ILF: indicates functionalities that use internal stored data
EIF: indicates functionalities that use external stored data (an EIF is an ILF for another app)
EI: Data retrieved from an input
EO: Data printed to show to an interface
EQ: Data asked but not computed

ILF and EIF compose the Data Funcionalities
EI, EO, EQ compose Transactions

In data functionalities are also computed DET (single-field user data) and RET (multiple-field user data)
In transacion are computed DET and FTR (entities involved in the transaction)

CoCoMo is the achronym of Constructive Most Model, it is a tool to estimate time and effort using these formulas:
        T = aSb    &  M = cTd
a,b,c,d are constants given by a table that change based on the difficulty of the project: from easy to difficult. 
It uses also project parameters that indicates the type of project we are going to develop in terms of language, technologies and soon.
e.g for an easy project we use:

        T = 2.4S1.05 & M = 2.5S0.38

CoCoMo II is the obvious evolution of CoCoMo driven by new sw life-cycles and more complex projects.
The new version is much more accurate, in fact it uses some Scale Factors and Effort multipliers to adapt the calculus of effort and time.

- Early design (is an estimation done before start):

CoCoMo II also give indications about SW Reuse.
The SW Reuse is one of the basic concept of today SW development: it is the perfect example for Legacy systems (System used for a purpose different from the one they were developed for); nowadays there is no reason  for develop a system from scratches (also the fundamental of 1-tier architectures), so we have to adapt the code  to our purposes, and for doing it, there are some parameters that help us calculating the effort needed to interpolate that code: 
SW Understanding: how much we understand about that project
Assessment & assimilation: how we will use it on our project
% of modification on the code: how much we are going to change on the code.
